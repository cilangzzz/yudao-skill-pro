# 注意事项

## 好友关系相关

### 1. 双向好友模型的事务一致性

- 好友关系采用双向模型，每对好友存储 2 行记录
- 创建好友关系时必须在同一事务中插入 2 行记录，否则会导致数据不一致
- 删除好友时必须同时删除 2 行记录
- 不要只查询一个方向就认为好友关系完整存在

### 2. 好友属性的独立性

- 每个方向的好友记录拥有独立的 silent/pinned/blocked/display_name 属性
- A 拉黑 B 不影响 B 对 A 的视角（B 不知道自己被拉黑）
- 更新好友属性时只更新当前用户方向的记录，不要误操作对方的记录

### 3. 好友请求的幂等性

- 同一对用户之间不应同时存在多个未处理的好友请求
- 发送好友请求前需检查是否已存在未处理的请求
- 同意好友请求后，需要检查是否已经是好友（防止重复创建关系）

## 群组管理相关

### 4. 群主退出限制

- 群主不可直接退出群组
- 群主退出前必须先转让群主身份或解散群组
- 解散群组时需要同时清理所有成员记录和相关数据

### 5. 管理员权限边界

- 管理员可以踢出普通成员，但不可踢出群主和其他管理员
- 只有群主可以设置/取消管理员
- 只有群主可以转让群主身份
- 权限校验时需要正确判断操作者和被操作者的角色关系

### 6. 全员禁言与个人禁言

- 全员禁言（muted_all）和个人禁言（mute_end_time）是独立的
- 判断成员是否可发言时，需要同时检查全员禁言和个人禁言状态
- 个人禁言有时间限制，全员禁言没有时间限制
- 群主和管理员通常不受禁言限制

### 7. 入群审批开关

- `im_group.join_approval` 控制是否需要审批
- 关闭审批时，申请入群直接成为成员，不创建请求记录
- 开启审批时，需要管理员/群主同意后才能加入

### 8. 置顶消息存储

- 置顶消息 ID 列表存储在 `im_group.pinned_message_ids` 字段中（JSON 数组）
- 添加/取消置顶时需要读取-修改-写入整个数组
- 注意并发操作可能导致数据覆盖，建议使用乐观锁或分布式锁

## 消息收发相关

### 9. clientMessageId 幂等发送

- `client_message_id` 是消息的幂等键，全局唯一
- 客户端生成 UUID 作为 clientMessageId，服务端通过唯一索引保证幂等
- 网络重试时携带相同的 clientMessageId，服务端检测到已存在则直接返回
- 不要使用可预测的值（如时间戳）作为 clientMessageId

### 10. 事务感知异步推送

- 消息推送使用 `TransactionSynchronizationManager.registerSynchronization(afterCommit)` 回调
- 确保数据库事务提交后再推送消息，避免接收方读到未提交的数据
- 推送失败不应影响消息发送的事务提交
- 推送逻辑应有重试机制

### 11. 已读游标的 Redis 存储

- 已读游标存储在 Redis 中，key 格式为 `im:read:cursor:{userId}:{targetId}`
- 私聊场景 targetId 为对方用户 ID，群聊场景 targetId 为群组 ID
- Redis 数据丢失会导致已读状态丢失，需要有降级方案
- 不要在每次消息拉取时都更新已读游标，避免 Redis 写入压力

### 12. 消息撤回的时间窗口

- 消息撤回有时间限制（通常为 2 分钟或 5 分钟）
- 超过时间窗口的消息不可撤回
- 撤回操作更新消息状态为"已撤回"，不物理删除消息
- 群聊中群主/管理员可以撤回其他人的消息

### 13. 敏感词过滤时机

- 敏感词过滤在消息发送时执行，不是在存储时
- 过滤后的消息内容直接存储，不再保留原始内容
- 敏感词库更新后，已发送的消息不受影响
- 过滤方式包括替换（用 * 替换）和拦截（拒绝发送）两种

## RTC 通话相关

### 14. LiveKit Webhook 安全

- `/im/livekit/webhook` 接口标记为 `@PermitAll` + `@TenantIgnore`
- 必须通过 JWT + SHA256 签名验证请求的合法性
- 不要在未验证签名的情况下处理 Webhook 事件
- LiveKit API Secret 泄露会导致 Webhook 被伪造

### 15. 通话状态一致性

- RTC 通话状态涉及多个参与者，状态变更需要考虑并发
- 最后一个参与者离开时应自动结束通话
- 超时未接听的通话需要定时任务清理
- 参与者状态为 INVITING 时，超时后应标记为 NO_ANSWER

### 16. 房间 Token 生成

- LiveKit 访问 Token 包含房间权限和用户身份信息
- Token 有过期时间，需要在加入房间前生成
- 不要缓存 Token 太久，避免过期导致加入失败
- Token 中的用户身份应与系统用户 ID 一致

## 敏感词相关

### 17. 敏感词库的内存占用

- 敏感词库基于 Trie 树结构，全部加载到内存中
- 敏感词数量过多时会占用较多内存
- 建议控制敏感词数量在合理范围内（万级以内）
- 敏感词更新后需要重建 Trie 树

### 18. 敏感词匹配模式

- 支持精确匹配和拼音匹配两种模式
- 拼音匹配依赖 pinyin4j 库，会增加处理耗时
- 空格和特殊字符可能会干扰匹配结果
- 建议在敏感词入库时进行规范化处理

## 频道相关

### 19. 频道广播的接收者管理

- 频道消息的 `receiver_user_ids` 字段存储接收用户 ID 列表
- 大量接收者时该字段会很长，建议控制在合理范围内
- 为空时表示广播给频道所有关注者
- 接收者列表的生成逻辑在业务层实现

### 20. 频道素材与消息的关联

- 频道消息通过 `material_id` 关联素材
- 删除素材前需要检查是否有关联的消息
- 素材内容使用 HTML 格式存储，注意 XSS 防护

## 通用注意事项

### 21. 错误码规范

- IM 模块错误码遵循 `1-009-xxx-xxx` 格式
- 常见错误码：
  - 好友已存在
  - 不是好友关系
  - 群组不存在
  - 不是群成员
  - 已被禁言
  - 消息已撤回
  - 撤回超时
  - 通话不存在

### 22. 租户隔离

- IM 模块的大部分表参与租户隔离
- LiveKit Webhook 接口使用 `@TenantIgnore` 忽略租户
- 敏感词表是否参与租户隔离取决于业务需求

### 23. 消息内容的 JSON 格式

- 消息 `content` 字段使用 JSON 格式存储
- 不同消息类型的 content 结构不同（文本/图片/语音/视频/文件）
- 解析 content 时需要根据 type 字段选择对应的解析逻辑
- content 字段长度有限制，大文件消息应只存储元信息，文件内容通过文件服务存储

### 24. 性能优化

- 群聊消息推送给大量群成员时注意推送性能
- 已读游标使用 Redis 而非数据库，避免频繁写入
- 消息列表查询建议使用 id 范围查询而非 offset 分页
- 大群（成员数多）的消息推送考虑使用批量推送和消息压缩
